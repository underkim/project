# ADR 0004: Python 의존성 관리 도구 선택

## Status
Accepted (2026-05-05)

## Context
Python 프로젝트는 패키지 의존성을 관리하기 위한 도구가 필요하다.
주요 요구사항은 다음과 같다.

- **재현성**: 동일한 환경을 다른 머신에서 재현 가능해야 한다 (lockfile)
- **격리**: 가상환경(venv)으로 시스템 Python과 분리되어야 한다
- **충돌 해결**: 트랜지티브 의존성 충돌이 자동으로 해결되어야 한다
- **개발 속도**: 설치 및 업데이트가 빠를수록 반복 주기가 단축된다
- **학습 가치**: 본 프로젝트는 학습 목적도 포함하므로, 최신 표준을 경험하는 기회로서의 가치도 평가 대상이다

후보 도구 비교:

| 항목 | pip + venv | poetry | uv |
|---|---|---|---|
| 설치 속도 | 보통 | 느림 | 매우 빠름 (Rust 구현) |
| Lockfile | 수동 (pip-tools 필요) | 자동 (poetry.lock) | 자동 (uv.lock) |
| 가상환경 통합 | 분리 | 통합 | 통합 |
| 학습 곡선 | 낮음 | 중간 | 낮음 |
| 생태계 성숙도 | 매우 높음 | 높음 | 빠르게 성장 중 |
| 출시 시점 | 2008 | 2018 | 2024 |

## Decision
**uv**를 선택한다.

전제: 의존성 관리 도구는 `pyproject.toml`이라는 표준 포맷을 공유하므로,
이후 다른 도구로의 교체 비용이 낮다. 따라서 익숙한 도구를 답습하기보다는
새로운 도구를 채택하여 학습 가치를 확보하는 편이 합리적이라 판단한다.

이유:
1. Rust 기반 구현으로 설치 속도가 압도적으로 빠르다
2. 가상환경, 의존성 잠금, 패키지 설치가 단일 도구로 통합되어 있다
3. FastAPI, SQLAlchemy 등 주요 패키지와의 호환성이 검증되어 있다
4. `pyproject.toml` 표준 준수로 향후 poetry, pip 등으로 이식이 가능하다
5. 2024년 등장한 최신 도구를 학습하는 것은 향후 다른 프로젝트에도 활용 가능한 자산이 된다

## Consequences
### 긍정
- 의존성 설치 시간이 단축되어 개발 반복 주기가 짧아진다
- venv, lock, install이 단일 명령어 체계로 통합되어 명령 학습 부담이 적다
- 최신 표준 도구를 경험하며 학습 가치를 얻는다
- `pyproject.toml` 표준 준수로 향후 다른 도구로 이식할 수 있다

### 부정
- 출시 시점이 비교적 최근(2024)이라 일부 엣지 케이스에서 자료 축적이 부족할 수 있다
- 빠른 버전 업데이트가 이뤄지므로 동작 변경 사항을 주기적으로 점검해야 한다
- 익숙하지 않은 도구이므로 초기 학습 비용이 발생한다
