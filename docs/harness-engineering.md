# 하네스 엔지니어링(Harness Engineering)이란

## 정의

**하네스(harness)** 는 LLM 자체가 아니라, LLM이 실제로 코드베이스에서 안전하고 효과적으로
일하도록 만드는 **주변 장치 전체**를 가리킨다: 어떤 도구를 쓸 수 있는지, 무엇을 자동으로
허용/차단할지, 세션이 시작될 때 무엇을 준비해둘지, 반복 작업을 어떻게 절차화할지, 실수를
어느 지점에서 잡아낼지. Claude Code 자체가 하나의 "하네스"이고, `.claude/` 아래의 설정은
그 하네스를 **이 프로젝트에 맞게 튜닝**하는 부분이다.

같은 모델이라도 하네스가 부실하면:
- 매 세션 의존성을 수동 설치해야 하고 (시간 낭비, 재현성 낮음)
- 매 명령마다 권한 승인을 받아야 해서 마찰이 크고
- "새 모듈 만들 때 이 순서로 해라" 같은 반복 지식이 대화마다 다시 설명돼야 하고
- 실수(깨진 테스트를 커밋, `.env` 노출 등)를 사람이 매번 리뷰로 잡아야 한다.

하네스 엔지니어링은 이런 반복·마찰·실수 지점을 **코드/설정으로 고정**해서, 에이전트가
매번 같은 실수를 할 여지를 줄이고, 사람의 개입 없이도 안전한 기본 동작을 하도록 만드는
작업이다.

## 구성 요소 6가지

### 1. CLAUDE.md — 영속적 메모리 / 지시사항

세션이 끝나면 대화 맥락은 사라지지만 `CLAUDE.md`는 매 세션 로드된다. "이 프로젝트는
FastAPI+Next.js고, SQLAlchemy async에서 lazy loading 하면 안 되고, AI 서비스는
session.begin()을 쓰면 안 된다" 같은, **한 번 배웠는데 또 실수하면 안 되는 지식**을 여기
넣는다. 이 프로젝트의 CLAUDE.md가 이미 상세히 잘 되어 있다 (기술 스택, 규칙, 데이터 모델,
"자주 발생하는 실수 방지" 표까지). 하네스 엔지니어링에서 CLAUDE.md는 **"무엇을 해야 하는가"**
를 담당하고, 아래 hooks/permissions는 **"실제로 그렇게 하도록 강제/보조하는가"** 를 담당한다
— 즉 CLAUDE.md만으로는 지켜지는지 보장이 안 되고, 강제 장치가 있어야 신뢰할 수 있다.

### 2. Permissions — 마찰 조절 + 안전 경계

`.claude/settings.json`의 `permissions.allow/deny/ask`는 두 가지 역할을 동시에 한다:
- **마찰 감소**: 읽기 전용이고 안전한 명령(`git status`, `uv run pytest` 등)을 매번 승인받지
  않도록 허용 목록에 등록
- **안전 경계**: `rm -rf`, `git push --force`, 임의 코드 실행(`python3 -c` 와일드카드 등)은
  **절대 allowlist에 넣지 않는다** — 이건 마찰이 아니라 안전장치이기 때문

허용 목록은 "쓸 것 같아서"가 아니라 **실제 사용 패턴에서 도출**하는 게 안전하다. 이 프로젝트는
세션 transcript를 스캔해서 실제로 반복 사용된 읽기 전용 명령만 추가했다 (`fewer-permission-
prompts` skill, 상세는 `docs/harness-current-state.md`).

### 3. Hooks — 생명주기 이벤트에 거는 자동화

Hook은 "이런 일이 생기면 이 스크립트를 실행해라"는 규칙이다. 대표적인 이벤트:

| 이벤트 | 시점 | 이 프로젝트에서의 용도 |
|---|---|---|
| `SessionStart` | 세션 시작 시 | 의존성 설치 (`uv sync`, `npm install`) |
| `PreToolUse` | 도구 호출 직전 (차단 가능) | `git commit` 전 테스트+타입체크, 실패 시 커밋 거부 |
| `PostToolUse` | 도구 호출 직후 | (미사용) 예: 파일 저장 후 자동 포맷 |
| `Stop` | 에이전트가 응답을 끝낼 때 | (미사용) 예: 세션 종료 시 요약 알림 |

Hook은 **memory(지시사항)로 못 하는 일**을 한다 — "커밋 전에 항상 테스트를 돌려라"는
CLAUDE.md에 적어도 강제되지 않지만(에이전트가 깜빡할 수 있음), PreToolUse hook은 실제로
그 도구 호출 자체를 가로채 차단할 수 있다.

Hook을 만들 때의 핵심 원칙 (이 프로젝트에도 적용):
- **멱등성** — 여러 번 실행해도 안전해야 함 (`uv sync`, `npm install`은 이미 멱등)
- **비대화형** — 사용자 입력을 기다리면 안 됨
- **범위를 좁혀라** — `PreToolUse`를 모든 Bash 호출에 걸지 않고 `if: "Bash(git commit*)"`로
  좁혀서, 관련 없는 명령에는 지연이 전혀 없도록 함
- **먼저 raw 명령으로 검증한 뒤 wrap** — 에러를 숨기는 `|| true`는 검증 후에만 추가
- **실패 주입 테스트** — "통과할 때 조용히 넘어가는지"뿐 아니라 "실패할 때 정말 차단하는지"도
  반드시 확인 (이 프로젝트의 `pre-commit-check.sh`는 실제로 깨진 테스트를 임시로 넣어서
  차단 동작을 검증했다)
- **훅 자체의 성능도 설계 대상** — `pre-commit-check.sh`는 이후 실시간 작업 로그
  (`.claude/state/activity-log.json`) 기능을 위해, 스테이징된 파일이 로그 파일 하나뿐일 때는
  pytest/tsc를 생략하도록 확장했다. 안전장치(전체 검사)와 사용성(빠른 체크포인트 커밋)이
  충돌할 때, "무엇이 바뀌었는지"로 분기해 둘 다 satisfy한 사례.

### 4. Skills — 절차 지식의 재사용

Skill은 "이런 종류의 요청이 오면 이 절차를 따라라"는 실행 가능한 문서다. 이 프로젝트의
CLAUDE.md에는 "새 모듈 추가 패턴" 체크리스트가 텍스트로 있었지만, 텍스트는 매번 다시
"읽고 해석"해야 한다. `.claude/skills/new-module/SKILL.md`로 만들면:
- "새 모듈 추가해줘" 같은 요청에 자동으로 매칭되어 로드됨
- 순서·주의사항(dialect 가드, RLS, 트랜잭션 규칙 등)이 매번 정확히 재현됨
- 프로젝트마다 다른 세부사항(이 프로젝트는 SQLite/Postgres dialect 분기가 필요하다는 것
  같은)을 일반 지식보다 우선해서 따르게 됨

일반적인 소프트웨어 엔지니어링의 "런북(runbook)"이나 "플레이북"과 같은 개념이라고 보면 된다.

### 5. Subagents — 격리된 전문 작업자

이 프로젝트는 아직 프로젝트 전용 subagent를 정의하지 않았지만, 개념은 알아둘 가치가 있다.
`.claude/agents/*.md`로 정의하면 특정 도구 집합·프롬프트·모델을 가진 전문 에이전트를 만들
수 있다 (예: "코드 리뷰만 하는 에이전트", "마이그레이션 안전성만 검토하는 에이전트"). 이
프로젝트 규모(단일 사용자 포트폴리오, 7개 모듈)에서는 범용 Explore/Plan subagent로 충분해
보류했다 — 하네스 엔지니어링은 항상 필요한 만큼만 무겁게 만드는 것이 원칙이다 (과설계 금지는
CLAUDE.md의 코드 작성 원칙과도 일치).

### 6. CI — 하네스의 바깥쪽 고리

`.github/workflows/ci.yml`은 "세션 밖"에서 동작하는 하네스다. 로컬/세션 hook이 놓치는 것
(Playwright e2e처럼 오래 걸려 커밋마다 돌리기엔 비싼 검사)을 push/PR 시점에 잡아준다.
좋은 하네스는 **로컬 hook과 CI가 같은 걸 이중으로 다르게 구현하지 않고, 같은 명령을
재사용**한다 (이 프로젝트의 pre-commit hook과 CI의 `test`/`typecheck` job은 정확히 같은
명령 `uv run pytest`, `npx tsc --noEmit`을 쓴다).

## 이 프로젝트에 적용한 결과

`docs/harness-current-state.md`에 파일 단위 인벤토리와 각 구성 요소의 검증 결과가 있다.
요약하면:

1. SessionStart hook — 원격 세션마다 반복되던 수동 `npm install`/`uv sync`를 제거
2. `.claudeignore` — CLAUDE.md가 문서화만 하고 있던 보안 경계를 실제로 존재하게 함
3. Permission allowlist 확장 — 실제 사용 패턴 기반 1개 항목 추가 (`npx tsc --noEmit`)
4. PreToolUse pre-commit hook — CI에서 떨어질 코드가 커밋되는 것 자체를 차단
5. `new-module` skill — 반복되는 스캐폴딩 절차를 실행 가능한 형태로 고정

## 다음에 고려할 만한 확장

- **PostToolUse 포맷터** — 이 프로젝트에 prettier/black 같은 자동 포맷터를 도입하면
  Write/Edit 직후 자동 정리 가능
- **Stop hook 알림** — 긴 작업 완료 시 사용자에게 알림 (현재는 대화 종료 메시지로만 안내)
- **git push 시점 검사** — 지금은 커밋 시점만 막는데, 여러 커밋을 쌓아 올린 뒤 한 번에
  push하는 워크플로우라면 push 시점 검사도 의미가 있을 수 있음 (다만 중복 실행 비용 고려)
- **eslint 훅** — 현재 `frontend/package.json`의 `lint` 스크립트는 자동화되어 있지 않음
- **프로젝트 전용 subagent** — 모듈 수가 늘어나 코드 리뷰 부담이 커지면 검토

이 모든 확장은 "필요해질 때" 추가하는 것을 권장한다 — 하네스도 과설계하면 그 자체가
유지보수 부담이 된다.
